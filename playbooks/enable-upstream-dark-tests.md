# Upstream `.dark` Tests Enablement Playbook

## Goal
Enable all upstream `.dark` tests by default, one test at a time, while keeping the main suite reliable.

## Working Rules
1. Enable exactly one skipped test in one `.dark` file per iteration.
2. Run tests immediately after enabling it.
3. If it fails, fix compiler/runtime behavior first when feasible.
4. If behavior is already correct and only the assertion is outdated, update the test expectation.
5. Record every learning in this file.
6. If blocked and guidance is needed, ask Paul and record both question and answer here.

## Iteration Loop (Single-Test)
1. Pick one skipped test (`skip=...`) from a `.dark` file.
2. Remove `skip` and replace with the most direct assertion (`= value`, `error="..."`, or `exit/stderr`).
3. Run full `./run-tests` to validate the local change.
4. Diagnose with `./dark --syntax=interpreter -r -e '<expr>'` when output/error text is unclear.
5. Resolve by:
   1. Compiler/runtime fix, or
   2. Test expectation update if runtime/compiler behavior is already correct.
6. Re-run full `./run-tests`.
7. Append results to the logs below.
8. Before landing a batch, ensure the latest run was full `./run-tests`.

## Default Rollout Gate
1. Keep default inclusion scoped while a file still has unresolved failures; use `./dark -r -e ...` probes for fast diagnosis.
2. When a file reaches the desired state (no unresolved failures for the enabled tests in that file), move it toward default execution by updating `src/Tests/TestRunner.fs` include logic.
3. Re-run full `./run-tests` after any `TestRunner` inclusion change.
4. Record the file-level rollout decision in the Learnings Log.

| --- | --- | --- | --- |
| Compile-error assertions fail because expected message text no longer matches | Full suite fails with `Expected error message '...' not found in stderr` while the failing case still exits non-zero during compile stage. | Treat this as diagnostic-text drift first: inspect current compiler error text, then update expectation strings if semantics are unchanged. Only change compiler behavior if message drift reveals a semantic regression. | Re-run full `./run-tests` and confirm no additional regressions. |
| Enabling one skipped case can reveal stale assertions nearby | After replacing one `skip=...` with a direct assertion, other assertions in the same area fail with message-mismatch errors. | Keep the newly enabled case, then update only the stale assertions whose semantics are unchanged. Do not weaken assertions to hide real regressions. | Re-run full `./run-tests` after each assertion update. |
| Deciding when to include a file by default | A file has no unresolved failures for the currently enabled cases and repeated full runs are stable. | Move that file into default inclusion in `src/Tests/TestRunner.fs`, while keeping unstable files opt-in until fixed. | Re-run full `./run-tests` immediately after inclusion changes. |
| `Builtin.testRuntimeError` in an executed branch should be asserted as runtime failure, not skipped | Probe the expression directly with `./dark -r -e ...`; if it compiles and exits with `Uncaught exception: <msg>`, branch execution is working. | Replace `skip=...` with `exit=1 stderr="Uncaught exception: <msg>"` for that case. Keep `skip` only where behavior is still unsupported. | Run full `./run-tests` and confirm no regressions outside the touched file. |
| If-condition rejects known runtime-failing expressions as `__runtime_error` | A condition like `if Builtin.testRuntimeError "msg" then ...` fails at compile time with `must be a Bool ... got a __runtime_error`. | In the typechecker’s `If` condition handling, keep the normal non-Bool path, but for known runtime-failing expressions re-check the condition with an expected `Bool` context so runtime-failure expressions can typecheck and fail at runtime. | Enable one such skipped condition assertion and run full `./run-tests` to ensure the new case passes and existing condition diagnostics remain stable. |
| `if <cond> then <expr>` keeps strict branch typing (implicit `else ()`) | A one-armed `if` with non-Unit `then` branch reports `if branches must have same type: expected <ThenType>, got Unit`, even when `<cond>` is a runtime-failing expression. | Treat this as expected static typing unless language semantics change. Enable the test with a compile-error assertion that checks the branch mismatch message. | Run full `./run-tests` and verify there are no regressions in other `if` typing diagnostics. |
| Boolean operators reject non-`Bool` operands before runtime-error propagation | Expressions like `true && Builtin.testRuntimeError "msg"` or `false || Builtin.testRuntimeError "msg"` fail at typechecking with `<op> only supports Booleans`. | Assert these with `error="<op> only supports Booleans"` rather than runtime-error assertions, unless short-circuiting and operand typing rules are intentionally changed in the compiler. | Run full `./run-tests` after updating assertions and confirm no regressions in boolean-op tests. |
| Upstream `.dark` expressions may require explicit `L` suffix for integer literals | A case with an unsuffixed integer (for example `5 && true`) fails before intended type assertions and does not match expected error text. | Use `Int64` literal spelling (`5L`) in compiler/interpreter expression cases when assertion intent is about operator typing rather than literal parsing. | Re-run full `./run-tests` and ensure the failure shifts to the intended assertion path. |
| Prefer `error="..."` for compile-time diagnostics in `.dark` assertions | A test uses `Builtin.testDerrorMessage` but failures show compile diagnostics prefixed with `Type error:` and mismatches even though the core message is correct. | For compile-time failures, switch assertions to `error="..."` so checks align with compiler diagnostics instead of runtime error helper formatting. | Re-run full `./run-tests` and confirm compile-error cases pass without weakening the message content. |
| Legacy upstream helper names can be missing in this runtime | Full suite fails with `Undefined variable` for helper names like `Builtin.testNan` or `Stdlib.Float.multiply`, but equivalent core expressions compile and evaluate correctly with `./dark --syntax=interpreter -r -e ...`. | Keep the semantic intent and rewrite to supported primitives/operators (for example `Builtin.testNan` -> `(0.0 / 0.0)` and `Stdlib.Float.multiply a b` -> `(a * b)`), then assert the same behavior in the rewritten form. | Re-run full `./run-tests` and confirm the file passes without introducing unrelated failures. |
| `match` + `Builtin.testRuntimeError` can fail in print insertion after optimization | Runtime failure cases report `LIR conversion error: Print: Unexpected operand type for string: Int64Const 0L`, often when a selected branch reduces to `()` after optimization. | In print insertion, derive print type from the actual return atom shape; if the atom is `Unit`, print as `TUnit` even when the original program type was non-Unit. | Re-run full `./run-tests` and verify runtime-error match cases now produce the expected runtime error message instead of LIR conversion failures. |
| Constructor arity mismatch in patterns should be treated as non-match, not ANF error | Failing cases show `ANF conversion error: Constructor '<name>' has no payload type in variant lookup` for patterns like `NoArgs _`. | In ANF pattern-binding paths, treat payload-on-nullary constructor patterns as arity mismatch non-matches with no payload bindings, consistent with pattern-comparison lowering. | Re-run full `./run-tests` and confirm such cases report non-exhaustive match behavior instead of ANF conversion errors. |
| Guard/pattern validation happens even for branches that appear unreachable | Cases with invalid guard variables or incompatible grouped bindings fail at compile time despite earlier branches that would match first. | Keep assertions compile-time (`error="..."`) for undefined guard variables, incompatible grouped bindings, and incompatible grouped literal types; do not expect runtime short-circuiting to suppress these diagnostics. | Re-run full `./run-tests` and confirm diagnostics are stable and explicit. |
| Some upstream syntax variants are not supported in interpreter-mode test parsing | A case fails at parse time even though equivalent language behavior works, often with parser-specific errors (for example interpolation + triple-quoted form). | Replace with the closest supported syntax that preserves the same semantic intent, then keep the same assertion target. | Probe with `./dark --syntax=interpreter -r -e ...` and then run full `./run-tests` to confirm only syntax form changed, not behavior intent. |
| Interpreter syntax issues must be fixed in `1_InterpreterParser.fs`, not `1_Parser.fs` | A failing `.dark` test only reproduces with `--syntax=interpreter`, while compiler-syntax parsing appears unaffected. | Apply lexer/parser changes in `src/DarkCompiler/passes/1_InterpreterParser.fs` (for example adding `$"""..."""` support) and add a focused parser unit test to lock behavior. | Run a filtered repro (`--filter=<file>`) to confirm the targeted fix, then run full `./run-tests`. |
| Legacy `_v0` stdlib names can fail in user-level function resolution even when canonical functions exist | A case fails with `Undefined variable: Stdlib.<Module>.<fn>_v0`, while `Stdlib.<Module>.<fn>` works. | Add `_v0` fallback in user/generic function lookup (not only intrinsic module lookup): try exact name, then stripped name, and apply the same sequence with optional `Stdlib.` prefix. | Re-run a focused filter for the failing file, then full `./run-tests` to ensure name-resolution changes do not regress unrelated calls. |
| Arithmetic with left-side `Builtin.testRuntimeError` should surface uncaught exception text, not generic operator type errors | Expressions like `(Builtin.testRuntimeError "...") + <expr>` fail with `Invalid operation '+' on types: __runtime_error`. | Treat known left-side `Builtin.testRuntimeError` as an immediate failing path for arithmetic typing and emit `Uncaught exception: <msg>` when the message is statically known. | Add/enable a focused infix case for left-side runtime-error usage and confirm it matches expected stderr, then run full `./run-tests`. |
| Upstream infix mismatch messages may require legacy builtin-style wording | A case like `5L + true` expects `Builtin.int64Add's 2nd parameter ...` but compiler emits generic type mismatch text. | In arithmetic type-checking, map the relevant mismatch to legacy parameterized wording (function name, parameter ordinal/name, expected type, actual type/value) so message contracts remain stable. | Re-run the targeted file and verify exact stderr text, then run full `./run-tests`. |
| E2E parser can leak indented RHS continuation lines into shared preamble | A `.dark` assertion uses `= <Constructor.Path>` on one line and provides the argument on the next indented line; subsequent tests fail with unrelated preamble parse/type errors (for example constructor arity errors in `analyzePreamble`). | Treat indented continuation lines as part of the previous test when RHS starts with an incomplete expectation head, instead of classifying them as standalone preamble lines. | Add a focused `E2EFormatTests` case for continuation parsing, run `./run-tests --filter=\"E2E Format Tests\"`, then run full `./run-tests`. |
| Module-scoped `.dark` files need module-level test-start detection, not “any indented line” | Parsed test names unexpectedly become field/continuation fragments (for example record-field lines), followed by preamble parse failures from malformed leftovers. | Restrict new test starts to top-level or current module indentation, and treat deeper indentation as continuation/definition context unless explicitly matched as a test boundary. | Add focused `E2EFormatTests` coverage for multiline/module indentation patterns, run `./run-tests --filter=\"E2E Format Tests\"`, then run full `./run-tests`. |
| Multiline `.dark` test expressions are not always parenthesized wrappers | Files fail with “Unclosed multi-line expression” or malformed source extraction when a multiline test starts with a plain identifier (for example constructor name) and only reaches `=` on later lines. | Start multiline accumulation for non-definition module-level lines even without leading `(`, and when multiline parsing is ambiguous, fall back to general separator-based parsing instead of forcing `) =` extraction. | Add focused parser tests for multiline pipe and non-parenthesized multiline forms, run `./run-tests --filter=\"E2E Format Tests\"`, then run full `./run-tests`. |
| Interpreter parser currently requires explicit commas between record fields | Target file parse fails in preamble with `Expected ',' or '}' after record field` when record type/record literal fields are newline-separated without commas. | Treat this as a syntax-support gap: either add comma-less field separator support in interpreter parser for both record type defs and record literals, or keep the file out of default rollout until that parser behavior is implemented. | Reproduce with file-level filter, then verify parser behavior with focused tests and re-run full `./run-tests`. |
| Multiline `let` expressions without explicit `in` can fail as `Expected expression` | Running a source shaped like `let x = ...` followed by newline body expression fails in interpreter syntax, especially in upstream lambda-heavy files. | Treat this as a newline-sensitive parser gap around `let` binding boundaries; either implement parser support for newline-delimited `let ... <body>` forms or keep the file opt-in until that syntax is supported. | Reproduce with `./dark --syntax=interpreter -r <file-or-expr>` and targeted `./run-tests --filter=<file>.dark`, then run full `./run-tests` after any parser change. |
